# 社交系统

> **定义**：构建玩家关系链，提升留存和传播。通过人与人之间的连接增加离开成本。
> **适用范围**：几乎所有多人在线游戏

---

## 1. 公会/社团系统

### 1.1 公会基础
| 要素 | 说明 | 设计要点 |
|---|---|---|
| **创建/加入** | 创建需消耗资源，加入需申请或自由加入 | 降低加入门槛，提高创建门槛 |
| **公会等级** | 成员活跃度贡献公会经验，升级解锁功能 | 等级要有明显成就感 |
| **公会建筑** | 公会大厅、科技院、商店、副本入口 | 建筑升级需要全公会贡献 |
| **公会科技** | 全员属性加成、资源产出加成 | 科技树设计要有选择性 |
| **职位体系** | 会长、副会长、精英、普通成员 | 权限分级，管理便捷 |

### 1.2 公会活动
| 活动 | 说明 | 设计要点 |
|---|---|---|
| **公会战** | 公会间对抗 | 实力相近匹配，奖励与贡献挂钩 |
| **公会副本** | 公会成员共同挑战 | 难度设计要适合多人 |
| **公会BOSS** | 限时挑战强力BOSS | 伤害排名奖励 |
| **公会任务** | 每日公会贡献任务 | 简单低门槛 |

### 1.3 公会设计约束
- **新公会保护**：新公会要有保护期，避免被大公会碾压
- **离线处理**：离线成员处理要人性化
- **解散/转让**：机制要完善，避免纠纷
- **成员变更异常处理**：设计公会玩法时必须考虑成员变更（加入、退出、被踢）导致的异常情况：
  - **退出/被踢后贡献归属**：成员退出后，其历史贡献是否保留？已解锁的公会科技/建筑是否回退？
  - **公会战/副本中途退出**：公会战或副本进行中，成员退出或被踢，其占位、伤害统计、奖励分配如何处理？
  - **奖励发放时成员变更**：按周期结算奖励（如周贡献奖、赛季排名奖）时，若成员在结算前退出，奖励归属谁？是否允许"蹭奖励后退出"？
  - **公会排行榜/赛季**：公会排名按什么口径计算（实时成员/历史成员）？成员频繁跳槽是否影响排名公平性？
  - **冷却期设计**：加入新公会后是否需要冷却期才能参与公会玩法？防止恶意跳槽套利

## 2. 好友系统

### 2.1 好友功能
| 功能 | 说明 | 设计要点 |
|---|---|---|
| **添加好友** | 搜索ID、附近玩家、战斗后添加、推荐列表 | 添加门槛要低 |
| **好友列表** | 在线状态、最近登录、亲密度、备注 | 列表上限随等级/VIP提升 |
| **亲密度** | 组队、送礼、聊天提升亲密度 | 亲密度等级有明确奖励 |
| **助战系统** | 借用好友的高战力角色助战 | 助战次数限制 |
| **友情点** | 互赠友情点，用于友情抽卡 | 每日互赠上限 |

### 2.2 好友设计约束
- **好友上限**：合理设置，太多管理困难，太少不够用
- **删除机制**：删除好友要方便，避免被骚扰
- **推荐算法**：等级相近、同时在线、共同公会

## 3. 聊天系统

### 3.1 频道设计
| 频道 | 用途 | 设计要点 |
|---|---|---|
| **世界频道** | 全服玩家可见 | 发言频率限制、防广告 |
| **公会频道** | 公会成员专属 | 活跃度展示、公告功能 |
| **队伍频道** | 当前队伍成员 | 实时战术沟通 |
| **私聊** | 一对一 | 屏蔽/举报功能必备 |
| **系统频道** | 系统公告、活动通知 | 不可关闭或可选关闭 |

### 3.2 聊天功能
| 功能 | 说明 | 设计要点 |
|---|---|---|
| **敏感词过滤** | 自动过滤不当言论 | 必备功能 |
| **举报机制** | 玩家可举报不当行为 | 举报后要有反馈 |
| **语音转文字** | 语音消息转文字 | 移动端重要 |
| **表情/道具** | 聊天气泡、表情、道具展示 | 增加趣味性 |

## 4. 排行榜系统

### 4.1 榜单类型
| 类型 | 说明 | 更新频率 |
|---|---|---|
| **战力榜** | 全服战力排名 | 实时 |
| **等级榜** | 角色等级排名 | 实时 |
| **竞技场榜** | PVP排名 | 赛季制 |
| **公会榜** | 公会综合实力排名 | 周/赛季 |
| **成就榜** | 成就点数排名 | 实时 |
| **财富榜** | 持有货币/资产排名 | 实时/周 |

### 4.2 榜单设计要点
| 要素 | 说明 | 设计要点 |
|---|---|---|
| **分层展示** | 本服/跨服/好友圈 | 让玩家有参照系 |
| **奖励区分** | 排名奖励要有区分度 | 但不能让头部垄断全部 |
| **新服保护** | 新服/新玩家独立榜单 | 保护期设计 |
| **防刷机制** | 检测异常数据波动 | 保证公平性 |

## 5. 师徒系统

### 5.1 师徒功能
| 功能 | 说明 | 设计要点 |
|---|---|---|
| **拜师/收徒** | 高等级玩家收低等级玩家为徒 | 等级差门槛（如20级以上） |
| **师徒任务** | 共同完成指定任务获得奖励 | 任务难度适合协作 |
| **出师奖励** | 徒弟达到一定等级后出师 | 双方获得丰厚奖励 |
| **名师榜** | 展示收徒数量、出师数量排名 | 增加高玩带新动力 |

### 5.2 师徒设计约束
- **防刷机制**：防止"假师徒"刷奖励
- **成长平衡**：徒弟成长速度不能因师徒系统失衡
- **解除机制**：师徒关系解除要便捷

## 6. 组队系统

### 6.1 组队功能
| 功能 | 说明 | 设计要点 |
|---|---|---|
| **快速匹配** | 选择副本后自动匹配队友 | 匹配时间控制在30秒内 |
| **招募板** | 队长发布招募信息 | 显示战力要求、目标副本 |
| **组队加成** | 组队有额外经验/掉落加成 | 鼓励社交玩法 |
| **队伍语音** | 内置语音或快捷语音指令 | 移动端尤为重要 |
| **跟随队长** | 队员可自动跟随队长移动 | 减少等人时间 |

### 6.2 组队设计约束
- **匹配算法**：考虑战力平衡和职业搭配
- **挂机惩罚**：挂机/消极游戏要有惩罚
- **队长权限**：踢人权限要合理，防止滥用

## 7. 分享/邀请系统

### 7.1 分享场景
| 场景 | 分享内容 | 奖励 |
|---|---|---|
| **抽卡结果** | SSR出货截图 | 少量货币 |
| **战力提升** | 战力突破里程碑 | 少量货币 |
| **排名成就** | 竞技场/排行榜名次 | 少量货币 |
| **邀请好友** | 邀请码/链接 | 双方都有奖励 |
| **活动页面** | 活动H5页面 | 分享后解锁奖励 |

### 7.2 邀请系统
| 要素 | 说明 | 设计要点 |
|---|---|---|
| **邀请码** | 老玩家分享邀请码 | 被邀请人输入后绑定关系 |
| **邀请奖励** | 被邀请人达到一定条件后发放 | 防止刷号 |
| **召回奖励** | 召回流失好友获得奖励 | 社交裂变手段 |

## 8. 设计检查清单

- [ ] 社交功能是否完善？是否覆盖了核心社交场景？
- [ ] 公会系统是否健康？小公会是否有生存空间？
- [ ] 聊天系统是否安全？敏感词和举报机制是否完善？
- [ ] 排行榜是否公平？是否存在刷榜可能？
- [ ] 社交奖励是否适度？是否能驱动社交行为？
- [ ] 社交压力是否适度？是否会给玩家造成负担？
