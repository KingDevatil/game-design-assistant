# Scene: SYS-requirement - 需求分析（结合项目自身）

## 场景标识
- **ID**: SYS-requirement
- **名称**: 需求分析（结合项目自身）
- **岗位**: 系统策划
- **优先级**: P0（Phase 1）

## 场景目的
输入设计目的说明书 + 竞品调研报告，结合项目自身现状（已有系统、技术限制、美术资源、运营阶段），输出结构化的需求分析文档，明确：做什么、不做什么、为什么做、怎么做、依赖什么、风险在哪。

## 处理逻辑

1. **需求拆解**
   - 将设计目的转化为可量化的功能需求
   - 区分：核心需求（Must Have）/ 重要需求（Should Have）/ 锦上添花（Nice to Have）

2. **项目适配分析**
   - 评估每个需求与项目现状的匹配度
   - 识别技术/美术/运营层面的实现障碍
   - 提出资源缺口和解决方案

3. **依赖关系梳理**
   - 明确本系统依赖哪些已有系统
   - 明确哪些未来系统会依赖本系统
   - 标注接口和数据流需求

4. **优先级排序**
   - 基于影响面和实现成本，给出功能优先级建议
   - 建议分阶段迭代方案（MVP → 完整版）

6. **输出待澄清问题**
   - 至少提出5个需要产品/技术/美术确认的问题

## 质量自检清单

- [ ] 所有核心需求都有明确的"为什么做"的解释
- [ ] 项目适配分析中，每个风险都有对应的缓解方案
- [ ] 依赖关系图中没有遗漏关键系统
- [ ] 待澄清问题 ≥ 5个，且覆盖了技术、美术、运营维度
- [ ] 优先级排序有明确的判断依据（影响面 × 成本）
- [ ] 输出中没有任何与项目现状明显矛盾的假设
